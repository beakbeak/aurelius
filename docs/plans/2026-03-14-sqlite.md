# SQLite Media Database Plan

## Overview

Add a SQLite database to persist metadata about files in the media library. On startup, scan the filesystem and reconcile with the DB to detect added, changed, moved, and removed files. This replaces the need to open audio files via FFmpeg for every `handleTrackInfo` request, and enables serving `handleDirInfo` from the DB instead of the filesystem.

## Resolved Questions

- **Hashing**: Partial hash (first+last 64KB + file size via SHA-256). Fast enough for startup scans, unique enough for move detection.
- **DB location**: `StoragePath/aurelius.db`.
- **Fragment files**: Treated like regular media files — indexed in the DB as tracks.
- **Attached images**: Embedded image metadata stored in a JSON column on tracks. Directory images still discovered at request time (cheap ReadDir).
- **Search index**: Eventually replace Bleve with SQLite FTS5. Out of scope for this plan.
- **Startup behavior**: Scan blocks startup initially. Background scanning can be added later.
- **ReplayGain**: Store all four combinations: track/album × preventClipping true/false. `handleTrackInfo` continues to return the `preventClipping=true` values for now.
- **SQLite driver**: `github.com/mattn/go-sqlite3`.

## Schema

```sql
CREATE TABLE tracks (
    id              INTEGER PRIMARY KEY,  -- ROWID alias
    dir             TEXT NOT NULL,         -- library path of parent directory (e.g. "Artist/Album")
    name            TEXT NOT NULL,         -- filename only (e.g. "01 Track.flac")
    mtime           INTEGER NOT NULL,      -- file modification time (Unix seconds)
    hash            BLOB NOT NULL,         -- partial content hash for move detection
    tags            TEXT NOT NULL DEFAULT '{}',  -- JSON object of tag key/value pairs
    attached_images TEXT NOT NULL DEFAULT '[]',  -- JSON array of {mimeType, size} objects
    metadata        TEXT NOT NULL DEFAULT '{}',  -- JSON object for additional metadata (see below)

    UNIQUE(dir, name)
);

CREATE INDEX idx_tracks_dir ON tracks(dir);
-- hash index deferred to fsnotify iteration; full scan matches in memory

CREATE TABLE dirs (
    path    TEXT PRIMARY KEY,  -- library path (e.g. "Artist/Album")
    parent  TEXT NOT NULL      -- parent library path (e.g. "Artist"; "" for top-level)
);

CREATE INDEX idx_dirs_parent ON dirs(parent);
```

### `metadata` JSON Structure

The `metadata` column stores audio properties that don't need their own columns:

```json
{
    "duration": 245.5,
    "bitRate": 320000,
    "sampleRate": 44100,
    "sampleFormat": "s16",
    "replayGainTrack": 0.85,
    "replayGainAlbum": 0.90,
    "replayGainTrackNoclip": 1.02,
    "replayGainAlbumNoclip": 0.97
}
```

ReplayGain fields store all four combinations of mode (track/album) and preventClipping (true/false):
- `replayGainTrack`: track mode, preventClipping=true
- `replayGainAlbum`: album mode, preventClipping=true
- `replayGainTrackNoclip`: track mode, preventClipping=false
- `replayGainAlbumNoclip`: album mode, preventClipping=false

All four are omitted when ReplayGain data is unavailable.

### Design Notes

- **`dir` + `name` vs full path**: Splitting allows efficient directory listing queries (`SELECT * FROM tracks WHERE dir = ?`) without string parsing. The full library path is just `dir || '/' || name` (or `name` when `dir` is empty).
- **`dirs` table**: Enables directory listing without filesystem access. The `parent` index supports `SELECT * FROM dirs WHERE parent = ?` for subdirectory listing.
- **`hash` column**: Stored as BLOB (raw bytes, not hex string) for compactness. Not indexed in this iteration; the full scan has all removed tracks in memory for move matching. A hash index can be added later for fsnotify-based move detection.
- **`tags` as JSON**: Flexible schema for arbitrary tag keys. SQLite's `json_extract()` can query individual tags if needed later (e.g., for FTS5 indexing).
- **`attached_images` as JSON**: Lightweight; avoids a separate table for a rarely-queried list.
- **No `size` column**: File size is incorporated into the partial hash. The mtime alone is sufficient for change detection during scanning.

## Scanning Algorithm

The scan process reconciles the filesystem with the database. It is designed to be **modular**: the diff/move-detection and metadata-collection steps are shared between the full startup scan and future `fsnotify`-based incremental updates.

### Pipeline Overview

```
Full Scan:        Walk FS → Diff+Moves → Collect Metadata → Apply
fsnotify batch:   Events  → Diff+Moves → Collect Metadata → Apply
```

The shared stages are **Diff+Moves**, **Collect Metadata**, and **Apply**. The difference is only in how the initial set of candidate added/removed/changed files is produced.

### Phase 1: Walk Filesystem (full scan only)

Walk the media library root (skipping hidden files/dirs, not following symlinks). For each entry:

- **Directory**: Record `(path, parent)` in an in-memory set `fsDirs`.
- **Track file** (matches `getFileType() == fileTypeTrack`; includes fragments): Record `(dir, name, mtime)` in an in-memory map `fsFiles` keyed by `dir + "/" + name`.

Symlinks: Check `DirEntry.Type()` and skip any symlink entries. This avoids all duplicate-path issues.

```go
// FSEntry represents a file found on the filesystem during scanning.
type FSEntry struct {
    Dir   string
    Name  string
    Mtime int64
}
```

### Phase 2: Diff and Detect Moves

This phase produces a `ChangeSet` from the filesystem state and DB state. It combines diffing with move detection so that hashing (cheap) happens here, deferring the expensive FFmpeg metadata collection to Phase 3.

#### Step 2a: Diff

For a **full scan**, compare `fsFiles` against all DB tracks (via `ForEachTrack`):

1. Iterate all DB tracks. For each:
   - If present in `fsFiles` with same mtime → unchanged, remove from `fsFiles`
   - If present in `fsFiles` with different mtime → add to `Changed`, remove from `fsFiles`
   - If not present in `fsFiles` → add to `Removed`
2. Everything remaining in `fsFiles` → `Added`

Similarly, diff `fsDirs` against the `dirs` table to find added/removed directories.

For an **fsnotify batch**, the diff comes directly from the events: created files → `Added`, deleted files → `Removed`, modified files → `Changed`. No FS walk or DB iteration needed.

#### Step 2b: Hash and Detect Moves

1. Compute partial hashes for all `Added` files (fast — no FFmpeg needed).
2. Build a hash → `[]Track` map from `Removed` entries (hashes already in DB).
3. Match `Added` ↔ `Removed` by hash:
   - Exactly one removed + one added with same hash → move
   - Multiple matches → ambiguous, treat as separate remove + add
4. Matched entries become `Moves`; remove them from `Added` and `Removed`.

```go
// ChangeSet describes the differences between the filesystem and database.
type ChangeSet struct {
    Added   []HashedFSEntry     // files on FS but not in DB (after move detection)
    Changed []FSEntry           // files in both but with different mtime
    Removed []Track             // files in DB but not on FS (after move detection)
    Moves   []Move              // removed+added pairs matched by hash

    AddedDirs   []Dir
    RemovedDirs []Dir
}

// HashedFSEntry is an FSEntry with a computed partial hash.
type HashedFSEntry struct {
    FSEntry
    Hash []byte
}

// Move represents a file that was moved/renamed.
type Move struct {
    OldDir  string
    OldName string
    NewDir  string
    NewName string
    NewMtime int64
}
```

### Phase 3: Collect Metadata

For each file in `Added` and `Changed`, open via `aurelib` (or `fragment.New` for fragment files) to extract: duration, bitrate, sample rate, sample format, replay gain, tags, attached image info.

For `Changed` files, also recompute the partial hash (since the file content changed).

`Moves` do not need metadata collection — they retain the existing DB metadata (unless the file also changed, but a simultaneous move+modify within a single scan interval is treated as a remove+add for simplicity).

```go
// ScannedTrack is an FSEntry enriched with metadata read from the file.
type ScannedTrack struct {
    FSEntry
    Hash           []byte
    Tags           map[string]string
    AttachedImages []AttachedImageInfo
    Metadata       TrackMetadata
}

// ScanResult contains the fully resolved changes, ready to apply.
type ScanResult struct {
    AddedTracks   []ScannedTrack
    ChangedTracks []ScannedTrack
    RemovedTracks []Track
    Moves         []Move
    AddedDirs     []Dir
    RemovedDirs   []Dir
}
```

### Phase 4: Apply Changes to Database

Within a single transaction:

1. **Moves**: `UPDATE tracks SET dir = ?, name = ?, mtime = ? WHERE dir = ? AND name = ?`
2. **Changed**: `UPDATE tracks SET mtime = ?, hash = ?, tags = ?, attached_images = ?, metadata = ? WHERE dir = ? AND name = ?`
3. **Added**: `INSERT INTO tracks (...) VALUES (...)`
4. **Removed**: `DELETE FROM tracks WHERE dir = ? AND name = ?`
5. **Dirs**: Insert new dirs, delete removed dirs

### Modular Interface

```go
// Scanner coordinates filesystem scanning and database reconciliation.
type Scanner struct {
    db       *DB
    rootPath string
}

// FullScan walks the entire filesystem, diffs against the DB, detects moves,
// collects metadata, and returns a ScanResult ready to apply.
func (s *Scanner) FullScan() (*ScanResult, error)

// ProcessChanges takes a pre-computed ChangeSet (e.g. from fsnotify events),
// collects metadata for added/changed files, and returns a ScanResult.
// This skips the FS walk and DB diff — the caller provides the ChangeSet directly.
func (s *Scanner) ProcessChanges(changes *ChangeSet) (*ScanResult, error)

// DetectMoves examines Added and Removed entries in a ChangeSet, computes
// hashes for Added files, matches them against Removed hashes, and populates
// the Moves field. Matched entries are removed from Added and Removed.
func (s *Scanner) DetectMoves(changes *ChangeSet) error

// Apply writes a ScanResult to the database in a single transaction.
func (s *Scanner) Apply(result *ScanResult) error
```

**Full scan** calls: `FullScan()` (which internally does walk → diff → `DetectMoves` → collect metadata) → `Apply()`.

**fsnotify batch** calls: build `ChangeSet` from events → `DetectMoves()` → `ProcessChanges()` → `Apply()`.

### Library Changes

1. Add `*mediadb.DB` field to `Library` struct.
2. In `NewLibrary`, open/create the DB and run a full scan before setting up routes.
3. Modify `handleTrackInfo` to serve from the DB instead of opening the audio file:

```go
func (ml *Library) handleTrackInfo(libraryPath string, w http.ResponseWriter, req *http.Request) {
    dir, name := splitLibraryPath(libraryPath)
    track, err := ml.db.GetTrack(dir, name)
    if err != nil || track == nil {
        http.NotFound(w, req)
        return
    }

    // Build result from DB fields
    // Still need: favorite status (from playlist cache), directory images (from filesystem)
    images := track.AttachedImages
    images = append(images, getDirectoryImageInfos(ml.libraryToFsPath(libraryPath))...)

    result := Result{
        Name:            track.Name,
        Duration:        track.Metadata.Duration,
        ReplayGainTrack: track.Metadata.ReplayGainTrack,
        ReplayGainAlbum: track.Metadata.ReplayGainAlbum,
        Tags:            track.Tags,
        AttachedImages:  images,
        BitRate:         track.Metadata.BitRate,
        SampleRate:      track.Metadata.SampleRate,
        SampleFormat:    track.Metadata.SampleFormat,
        Dir:             ml.libraryToUrlPath("dirs", cleanLibraryPath(dir)),
    }
    // ... favorite check, writeJson
}
```

4. Modify `handleDirInfo` to serve from the DB instead of `os.ReadDir`:

```go
func (ml *Library) handleDirInfo(ctx context.Context, dirLibraryPath string, w http.ResponseWriter) {
    dirLibraryPath = cleanLibraryPath(dirLibraryPath)

    subdirs, err := ml.db.GetSubdirs(dirLibraryPath)
    // ... handle error

    tracks, err := ml.db.GetTracksInDir(dirLibraryPath)
    // ... handle error

    result := Result{
        Url:      ml.libraryToUrlPath("dirs", dirLibraryPath),
        TopLevel: ml.libraryToUrlPath("dirs", ""),
        Parent:   ml.libraryToUrlPath("dirs", cleanLibraryPath(path.Dir(dirLibraryPath))),
        Path:     dirLibraryPath,
        Dirs:     make([]PathUrl, 0, len(subdirs)),
        Tracks:   make([]PathUrl, 0, len(tracks)),
    }

    for _, d := range subdirs {
        result.Dirs = append(result.Dirs, PathUrl{
            Name: filepath.Base(d.Path),
            Url:  ml.libraryToUrlPath("dirs", d.Path),
        })
    }

    favorites, _ := ml.loadFavorites()
    for _, t := range tracks {
        trackPath := joinLibraryPath(t.Dir, t.Name)
        pu := PathUrl{
            Name: t.Name,
            Url:  ml.libraryToUrlPath("tracks", trackPath),
        }
        if favorites != nil {
            pu.Favorite = favorites.data.LineSet()[trackPath]
        }
        result.Tracks = append(result.Tracks, pu)
    }

    writeJson(ctx, w, result)
}
```

Note: Playlists are removed from `handleDirInfo` results (per plan to eventually remove playlist support). Both fragment descriptor files and their source audio files are stored in the `tracks` table. The existing behavior of hiding source files when fragments exist must be preserved: when building `handleDirInfo` results, if any fragment references a source file in the same directory, that source file is excluded from the track list. This can be done by checking fragment naming conventions (`fragment.GetSourceFile`) against the returned track names.

5. **`handleTrackStream`** and **`handleTrackImage`**: Still access the filesystem directly.

### What Doesn't Change

- Stream handling (still opens real files for transcoding)
- Image serving (still reads real files)
- Favorite management (still uses playlist cache / M3U files)
- Authentication, routing structure

## Partial Hash Implementation

```go
const hashChunkSize = 64 * 1024 // 64KB

func computePartialHash(path string) ([]byte, error) {
    f, err := os.Open(path)
    if err != nil {
        return nil, err
    }
    defer f.Close()

    info, err := f.Stat()
    if err != nil {
        return nil, err
    }

    h := sha256.New()

    // Write file size into hash
    binary.Write(h, binary.LittleEndian, info.Size())

    // Read first chunk
    buf := make([]byte, hashChunkSize)
    n, _ := io.ReadFull(f, buf)
    h.Write(buf[:n])

    // Read last chunk (if file is large enough that it doesn't overlap)
    if info.Size() > int64(hashChunkSize)*2 {
        f.Seek(-int64(hashChunkSize), io.SeekEnd)
        n, _ = io.ReadFull(f, buf)
        h.Write(buf[:n])
    }

    return h.Sum(nil), nil
}
```

   - Schema creation / migration

   - Update `handleTrackInfo` to read from DB
   - Update `handleDirInfo` to read from DB

   - Unit tests for scan/diff/move logic
